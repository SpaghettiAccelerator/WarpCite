# WarpCite — Agent Instructions

You are working in a workspace that uses the WarpCite VSCode extension. The
user's goal is to quickly write an academic document in Typst (a paper or a
thesis such as a Bachelorarbeit). Your job is to help write that document with
correct, verifiable citations.

## Source layout

- Every source lives in its own folder: `sources/<source-name>/` containing the
  PDF, an optional `<pdf-name>.ocr.json` (Mistral OCR result) and a `source.bib`
  (BibTeX entry for this source, citation key = folder name).
- `sources.bib` next to the `sources/` folder is generated by WarpCite: it
  aggregates all `sources/*/source.bib` files between
  `@Comment{WarpCite: BEGIN autogenerated ...}` / `@Comment{WarpCite: END autogenerated}`
  markers. Never edit between the markers; content above them is kept.

## Citations in Typst

Citations use the citation function configured in the `warpcite.citationFunction`
setting (default `dhbwCite`):

```typst
#dhbwCite("<source-name>", page: <printed page>, highlight: (<from>, <to>))
```

- `<source-name>` is the folder name under `sources/`.
- `page` is the **printed** page number as shown on the page itself — take it
  from the `page` field of the corresponding page object in the `.ocr.json`
  (a string; can be a roman numeral like `"iv"` for front matter).
- `highlight` is the character range `(from, to)` inside that page's `markdown`
  string in the `.ocr.json` that contains the cited passage.

## How to research and cite

1. **Every externally sourced statement must be cited.** Anything taken from
   the literature — theoretical introductions, definitions, methods, figures,
   numbers, claims — needs a citation. Only the user's own contribution and
   internal/company knowledge are exempt.
2. **Research exclusively from the OCR'd sources in this workspace.** The
   `sources/*/*.ocr.json` files are your only permitted source of cited
   information. Read their per-page `markdown` to find material. Do NOT cite
   from your own training knowledge, the web, or the PDF binary — if the
   information is not in an `.ocr.json`, do not invent a citation; If you cant cite
   something from a `.ocr.json` you do not write it!
3. **Reference as precisely as possible.**
   - Always set `page:` to the printed page number of the page that contains
     the passage (the `page` field, not the array index).
   - Whenever you can locate the exact passage, also set
     `highlight: (from, to)` — the character offsets into that page's raw
     `markdown` string that span the cited passage. This lets the user see
     exactly where the information comes from via hover and the preview panel,
     so include it whenever possible.
   - Verify offsets against the actual `markdown` string of that page; do not
     estimate them.
   - Cite via the dhbwCite function defined in Typst.
4. **Place citations next to the claims they support**, one citation per
   claim — do not collect them at the end of a paragraph if they belong to
   individual statements.

## Writing style guidelines

All document text is formal German academic prose (Wissenschaftsdeutsch) at
DHBW-Projektarbeit/Thesis level. Output is German unless instructed otherwise.

**Register and voice**

- Never use first person ("ich", "wir") and never address the reader. Refer to
  the document as "die vorliegende Arbeit", "diese Projektarbeit" or "die Arbeit".
- Default to passive voice and impersonal constructions ("wird ... vorgenommen",
  "erfolgt eine Betrachtung", "es empfiehlt sich") and nominal style ("die
  Erhebung und Analyse von Anforderungen", "zur Steigerung der Transparenz").
- Tone is sober, assertive and didactic: state claims as established knowledge
  backed by a citation. No hedging ("vielleicht", "eventuell") — either assert
  with a citation or scope explicitly (see below).

**Chapter and section structure**

- Open every theoretical chapter with: (1) a 1–2 sentence relevance claim
  ("... ist unerlässlich", "spielt eine entscheidende Rolle", "wesentliches
  Instrument"); (2) an immediate, cited definition of the central term ("kann
  als ... verstanden werden"); (3) a roadmap paragraph announcing the chapter's
  structure ("Zunächst ... Anschließend ... So wird aufgezeigt, wie ...").
- Subsections repeat relevance-then-definition in miniature, without a roadmap.
- Reference earlier chapters explicitly to build a cumulative argument ("wie
  sie in @kap:grundlagen erläutert wurden").
- Refer back to previously introduced terms, definitions, models and results
  whenever it is needed and rational to do so ("das in Kapitel 3 eingeführte
  Kano-Modell", "die zuvor definierten Anforderungskategorien") instead of
  re-explaining them or leaving the connection implicit.
- Reference chapters, sections, figures, tables and equations with Typst's
  native referencing, never with hard-coded numbers. Attach a label to the
  element (`= Anforderungsmanagement <kap:anforderungsmanagement>`,
  `#figure(...) <fig:pdca>`) and reference it with `@kap:anforderungsmanagement`
  or `@fig:pdca` in the text, so numbering stays correct when the document
  changes. The German supplement can be forced with brackets where the
  automatic one does not fit ("@kap:anforderungsmanagement[Kapitel]"). Add
  labels to the headings, figures and tables you create so they are
  referenceable later.
- When limiting depth or breadth, scope openly and justify it: "Die im
  Folgenden dargestellten ... werden lediglich auf einer übergeordneten Ebene
  behandelt ..., um den Umfang der Projektarbeit in einem angemessenen Rahmen
  zu halten."

**Paragraphs and sentences**

- Paragraphs have 3–6 sentences and one topic each, following claim →
  elaboration → synthesizing closing sentence ("So ...", "Damit ...", "Auf
  diese Weise wird ...", "Durch diese ... trägt ... wesentlich dazu bei, ...").
- Write long, hypotactic sentences (typically 20–35 words) with subordinate,
  relative and participial constructions; avoid staccato sequences of short
  sentences.
- Connect sentences and paragraphs with conventional connectives: "Darüber
  hinaus", "Im Gegensatz dazu", "Vor diesem Hintergrund", "Auf dieser Basis",
  "Dabei", "Hierbei", "Daran schließt ... an".
- Avoid em-dashes, semicolons and colons inside prose sentences. Every sentence
  must be complete in itself without such markers. Instead of splicing two
  thoughts with punctuation, write two full sentences joined by a connective.
  (The em-dash in table/figure captions and bold run-in headings is a
  formatting convention and remains the only exception.)

**Terminology**

- Define every technical term before using it, ideally as a copular definition
  ("Ein System ist eine Einheit von Komponenten, die ...").
- Introduce norms and concepts with the full official title first, abbreviation
  in parentheses, then use the abbreviation ("der Plan-Do-Check-Act-Zyklus
  (PDCA-Zyklus)"); cite norms with their full version designation at first
  mention ("ISO/IEC/IEEE 29148:2018-11").
- Attribute classifications and models to their named authors in-text
  ("Hruschka klassifiziert Anforderungen in zwei Kategorien: ...").
- Keep established English technical terms untranslated (Stakeholder, Use-Case,
  Requirements Engineering), embedded in German syntax.

**Citation density**

- Cite extremely densely: virtually every substantive sentence in a theoretical
  chapter carries a citation with its specific page — it is normal for every
  sentence of a paragraph to be cited. Consecutive sentences drawing on the
  same source are each cited individually with their page. Several citations on
  one sentence are acceptable when it synthesizes multiple sources. Only
  metadiscourse (roadmaps, scoping statements, transitions) goes uncited.

**Structured content**

- Prose dominates; no bullet-point-heavy body text. For enumerable aspects use
  bold run-in headings followed by 2–4 cited sentences ("**Isolation und
  Sandboxing** — Code muss in einer eingeschränkten Umgebung ausgeführt werden,
  die ..."); narrate classifications in prose with ordinal framing ("Die
  Basisfaktoren bilden die erste Kategorie ... In der zweiten Kategorie ...").
- Tables and figures carry numbered captions with em-dash and a cited source
  ("Tabelle 2 — Use-Case-Schablone") and are referenced in the surrounding text.
- At most one everyday analogy per chapter, immediately grounded and cited.
  Introduce concrete examples with "Beispiele hierfür sind ..." or "wie z.B.".
  Use German quotation marks („...") for literal terms and keywords.

**Avoid**

- First person, rhetorical questions, colloquialisms, contractions, marketing
  language, unexplained abbreviations, uncited factual claims. Superlatives are
  limited to the relevance framing of chapter openers.

## What you must not do

- Do not fabricate citations, and do not cite a source or page that does not
  actually contain the claim.
- Do not edit the autogenerated region of `sources.bib`.
- Do not modify `*.ocr.json` files by hand.
