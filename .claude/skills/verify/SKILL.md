# Verifying WarpCite

WarpCite is a VSCode extension (surface = the extension host). Verify by launching a real VSCode via `@vscode/test-electron`, not by unit-calling functions.

## Build

```bash
npm install && npm run compile   # emits out/extension.js
```

## Run against a real VSCode

1. In a scratch dir: `npm init -y && npm i @vscode/test-electron`.
2. Write a `runTest.js` that calls `runTests({ extensionDevelopmentPath: <repo root>, extensionTestsPath: <suite.js>, launchArgs: [<temp workspace>, "--disable-workspace-trust", "--disable-gpu", "--user-data-dir", <SHORT temp dir>] })`.
3. `suite.js` exports `run()`; it executes inside the extension host and can poll the filesystem / use the `vscode` module. Throw to fail, return to pass.

## Gotchas

- **Must pass a short `--user-data-dir`** (e.g. `fs.mkdtempSync(path.join(os.tmpdir(), "wc-"))`). The default lands under the long scratchpad path and VSCode's IPC socket exceeds the 103-char macOS unix-socket limit → `listen EINVAL` at startup.
- Build the test workspace as a temp copy of `example/` — the extension writes `source.bib` files into whatever workspace it opens; don't run it on the real repo unless you want those files created in `example/sources/`.
- First run downloads VSCode (~270 MB) into `.vscode-test/` in the scratch dir; later runs reuse it.
- A sample PDF for fixtures lives at `example/sources/examplesource1/1512.03385v1.pdf`.

## Packaging

`npm run package` (scripts/package.js) esbuild-bundles src → dist/extension.js, temporarily flips `main` to the bundle, runs `vsce package --no-dependencies`, and restores `main` to `out/extension.js` (dev/tests need the unbundled tsc output for require-cache stubbing). To verify a VSIX: install it into an isolated `--extensions-dir` via the VSCode CLI (`resolveCliArgsFromVSCodeExecutablePath`), then run a smoke suite from a dummy dev extension pointing at the same dirs — checks activation, commands, shipped AGENT_INSTRUCTIONS.md, watchers, and the bundled pdf.js path.

## Flows worth driving

- `source.bib` stubs are NOT auto-created (no startup scan, no pdf watcher, no delete-restore) — only the explicit `warpcite.generateBibtex` command ("Generate BibTex" in the pdf context menu) creates one, prompting before replacing an existing file.
- Generate BibTex runs a cascade (`out/bibExtract.js`, vscode-free): identifier hunt (filename arXiv id → pdf.js text layer/metadata/link annotations) → canonical BibTeX from arxiv.org / doi.org (network!) → if no identifier, a consent popup (`confirmAnnotationOcr` in `out/ui.js` — MUST be stubbed in tests, VSCode test mode throws "DialogService: refused to show dialog" on modals) gates a Mistral OCR `documentAnnotationFormat` call on pages [0,1] → declined/failed ⇒ empty TODO stub. `extractViaMistralAnnotation` on `out/bibExtract.js` is also stubable via the require cache for no-network cascade tests. Gotcha: pdf.js v3 in the Electron extension host needs `globalThis.pdfjsWorker = require("pdfjs-dist/legacy/build/pdf.worker.js")` or getDocument throws (isNodeJS is false under Electron). An image-only pdf fixture can be made via `sips -s format pdf tiny.png`.
- Aggregate: `sources.bib` next to `sources/` is created/updated automatically from all `sources/*/source.bib` inside `@Comment{WarpCite: BEGIN/END autogenerated}` markers; manual content above the block must survive regeneration, the block must be replaced (never duplicated), and deleting `sources.bib` regenerates it.
- `warpcite.runMistralOcr` ("Run Document OCR" in the pdf context menu) uploads the pdf to Mistral and writes `<name>.ocr.json` next to it (replace prompt if it exists — stub `confirmReplace` via `require(<repo>/out/ui.js)`, which is shared through the require cache; patching `vscode.window` does NOT work across module instances, and headless VSCode auto-cancels real modals). The API key lives in the `warpcite.mistral.apiKey` SETTING (no key file anymore) — for tests, read the user's key from `~/Library/Application Support/Cursor/User/settings.json` (JSONC!) and inject it via `getConfiguration("warpcite").update("mistral.apiKey", key, Global)` at suite start. Real API call, occasionally fails transiently. Each page gets a string `page` field parsed from the footer (`out/pageNumber.js` is vscode-free and unit-testable directly).
- The Typst citation function name is the `warpcite.citationFunction` setting (default `dhbwCite`) — hover, links, and preview all follow it.
- Hover in `.typ` files over `#dhbwCite("<source>", page: n, ...)` renders the OCR markdown of that printed page from `**/<source>/*.ocr.json` (first page if no `page:` arg). Drive it with `vscode.executeHoverProvider`.
